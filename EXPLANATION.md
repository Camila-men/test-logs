# Пояснительная записка

## Что реализовано

1. Регистрация (`/register`) и вход (`/login`) по email + паролю.
2. Каждая попытка входа и регистрации пишется в таблицу `auth_logs`: действие
   (`login`/`register`), статус (`success`/`error`), `user_id` (если известен), IP,
   User-Agent, текст ошибки (если есть), время.
3. `/logs` — список логов с пагинацией (20 на страницу) и фильтрами по всем полям
   таблицы, кроме текста ошибки; доступен только авторизованным пользователям.

## Почему так

### Сессии

Задание описывает именно веб-формы на Twig, а не отдельный клиент, которому нужен stateless-токен. Symfony Security «из коробки» даёт для
этого сценария готовый, проверенный временем механизм: CSRF-защита форм, безопасное
хранение сессии, `PUBLIC_ACCESS`/`ROLE_USER` в `access_control`.

### Логирование через события Security, а не внутри контроллера логина

`AuthenticationLogSubscriber` подписан на `LoginSuccessEvent` и `LoginFailureEvent` —
события, которые Symfony диспатчит независимо от того, какой `Authenticator` их
вызвал. Это разводит две ответственности: `AppAuthenticator` отвечает только за
проверку логина/пароля, а логирование — отдельный сервис (`AuthLogger`), которым
пользуется и подписчик (для входа), и `SecurityController::register()` (для
регистрации).

### `auth_logs` — отдельная таблица с индексами по фильтруемым полям

`action`, `status`, `ip`, `created_at`, `user_id` проиндексированы — именно по ним
идёт фильтрация на `/logs`. Поле `error_message` (`LONGTEXT`)
специально не индексируется и не участвует в фильтре — по условию задания фильтр по
нему не нужен, а текстовый поиск по такому полю потребовал бы отдельного решения, что избыточно для задачи.

`user_id` сделан nullable с `ON DELETE SET NULL`: неудачная попытка входа с
несуществующим email не имеет пользователя, но лог всё равно должен сохраниться.

### Пагинация — `Doctrine\ORM\Tools\Pagination\Paginator`, без дополнительных пакетов

Это часть `doctrine/orm`, которая уже нужна проекту как ORM — отдельный пакет вроде
`knplabs/knp-paginator-bundle` добавил бы ещё одну зависимость ради того же
`LIMIT`/`OFFSET` + `COUNT`, который `Doctrine\ORM\Tools\Pagination\Paginator` даёт
из коробки и корректно считает при `JOIN`.

### Bootstrap через CDN, без Webpack Encore/Node

Задание разрешает «любой удобный frontend-фреймворк» — Bootstrap подключён `<link>`
в `base.html.twig`. Отдельный шаг сборки ассетов (Node, Webpack Encore/AssetMapper)
усложнил бы разворачивание проекта (ещё один рантайм и ещё один build step в Docker)
без пользы для текущего объёма вёрстки — двух форм и одной таблицы.

### Хранение паролей и CSRF

- Хэширование пароля — `security.password_hashers` с `algorithm: auto` (Symfony сам
  выбирает лучший доступный алгоритм, сейчас это bcrypt/argon2i в зависимости от
  окружения) через `UserPasswordHasherInterface`.
- CSRF включён глобально (`framework.csrf_protection: true`); форма логина использует
  ручной скрытый `_csrf_token` (стандартный подход для `Authenticator`-компонента, не
  завязанного на Symfony Form), форма регистрации — CSRF-токен формы генерируется
  автоматически Symfony Form.
